# Wayland Siri: Full Analysis of 27 Open-Source Foundation Candidates

## TL;DR
- **Build your voice frontend on Newelle (GTK4/GNOME) and delegate execution to gptme**, with **agent-sh/computer-use-linux** as the Wayland-control MCP and **Agent Skills/SKILL.md** files as your procedure RAG — this trio matches every one of your requirements with the least custom glue. Newelle's GitHub README confirms it already ships a local voice loop ("🗣 Voice support... With wakeword support"), Ollama support, MCP, and "✨ Skills: Newelle supports skills from Claude, Codex..." — making it the single closest existing thing to your vision.
- **Cut everything that is cloud-first, dead, or architecturally mismatched**: AI Shell (stale, OpenAI-only), Leon (mid-rewrite, no usable 2.0), OVOS (smart-speaker stack, heavy), Seeva/azibom/Local-Voice (toy-scale), and the heavyweight Manus-style agents (AgenticSeek, LocalAGI, QwenPaw) that want 14B+ models or Docker stacks that bust a 6GB GPU.
- **Keep 2–3 finalists for the foundation (Newelle + gptme + optionally RealtimeVoiceChat/GLaDOS as a voice-loop reference), and treat the rest as building-block components** (computer-use-linux, AIChat, Fabric, Aider) or cut. Your hardware budget (6GB VRAM) is the binding constraint: it forces a 4B model fully on GPU with STT/TTS on CPU, which rules out anything assuming a 14B+ local model.

## Key Findings

**1. Your verified component picks are sound and current (June 2026).** Supertonic released "Supertonic 3" on 2026-04-29 (31 languages, ~99M params, ONNX, MIT code / OpenRAIL-M weights, runs real-time on CPU). Qwen3.5-4B is real and available via Ollama (`ollama run qwen3.5:4b`, 3.4GB Q4_K_M, 256K context, Apache-2.0, native tool calling), though there were documented Ollama tool-calling reliability bugs in some versions (issues #14493/#14745) — pin a known-good Ollama build. agent-sh/computer-use-linux is exactly the Wayland-first desktop-control MCP your architecture needs (AT-SPI + GNOME Shell extension + portals + ydotool), validated on Ubuntu 25.10/GNOME 50/Wayland.

**2. There is no single project that is already a "Wayland Siri."** Every candidate is either (a) a chat/desktop assistant with a voice layer but weak real desktop control (Newelle, NyarchAssistant), (b) a powerful agentic executor with no GNOME-native voice frontend (gptme, Goose, Open Interpreter, Cline), or (c) a voice loop with no agentic execution (RealtimeVoiceChat, GLaDOS, Local-Voice). Your architecture (local voice frontend + delegated executor + Skills RAG) is correct precisely because no one project spans all three layers.

**3. Newelle is the strongest single foundation for the frontend.** It is a native GTK4/libadwaita GNOME app (the exact target stack), GPL-3.0, ~1.37k stars, very actively maintained (v1.4.0 "Interfaces" shipped May 2026; version 1.0 released July 29, 2025 with mini apps, integrated browser, file manager/editor and programmable prompts per Grokipedia's release history; v0.8.0 with long-term memory shipped March 2025). It already supports: local models via Ollama and llama.cpp; voice chat with STT/TTS and **wakeword + call mode**; MCP tools and tool search; **Skills from Claude/Codex** (the exact Agent Skills/SKILL.md pattern you want); terminal command execution with a permission/safety model; an OpenAI-compatible API and a GUI API so external interfaces can drive it. Its weakness for your use case: its "desktop control" is terminal-command execution and mini-apps, not true Wayland GUI automation — you'd bolt computer-use-linux onto it via MCP.

**4. The best "hands" (executor) for a 6GB box is gptme.** MIT, ~4.3k stars (per the bradAGI/awesome-cli-coding-agents directory: "gptme ⭐ 4.3k — AI agent in your terminal with support for persistent autonomous agents. Runs code, edits files, browses the web."), very active — the gptme/gptme releases page confirms dev builds through "gptme version v0.31.1.dev20260423," with v0.31.0 (background jobs, cost tracking) shipped Dec 2025. It runs local via Ollama, executes shell, runs Python, patches files, browses via Playwright, and crucially ships a `computer` tool for full desktop GUI/computer-use plus subagents and MCP. It is lightweight Python (no Electron, no 14B assumption), which fits your constraints far better than Goose (Rust, heavier, ACP-centric) or Open Interpreter (good OS mode but AGPL/Apache nuance and heavier).

**5. The OpenClaw/Moltis/QwenPaw "personal agent gateway" class is the wrong shape for you.** OpenClaw (MIT, ~378k stars verified per bradAGI/awesome-cli-coding-agents: "OpenClaw ⭐ 378k — The original personal AI assistant you run locally; CLI with onboarding wizard, skills, tools, and multi-channel support (WhatsApp/Slack/Discord). MIT," created by Peter Steinberger) is a Node.js *messaging-channel gateway* with browser automation via Chrome DevTools Protocol and voice wake on macOS/iOS/Android — not a Linux/Wayland desktop driver, and it assumes always-on server use with cloud or large models. Moltis (Rust, ~2.7k) is an OpenClaw-inspired single-binary agent server, also channel-centric. QwenPaw (Apache-2.0, ~17.4k) is Alibaba AgentScope's chat-app-connected assistant with Skills and Ollama support but oriented to DingTalk/Feishu/QQ and cloud Qwen. All three are impressive but optimize for "reach me on my phone," not "drive my GNOME desktop by voice."

**6. Shell helpers (AIChat, Fabric, ShellGPT) are components, not foundations.** AIChat (Rust, ~10–29k stars depending on source, Apache-2.0/MIT) is an excellent all-in-one local LLM CLI with shell assistant, RAG, function-calling, and an OpenAI-compatible server — a great utility brick. Fabric (~42k) is a prompt-pattern framework with Ollama support and an Ollama-compatible server. ShellGPT (~12.1k, MIT, maintained, v1.5.1) does NL→shell via LiteLLM/Ollama but explicitly "not optimized for local models." None do voice or desktop control.

## Recommendations

**Stage 1 — Lock the foundation trio (build now):**
1. **Frontend = Newelle.** Fork or extend it. You inherit GTK4/GNOME-native UI, Ollama, voice loop with wakeword/call mode, MCP, Skills (Claude/Codex format = your SKILL.md RAG), and a permissioned command executor. This collapses months of frontend work.
2. **Executor = gptme**, invoked by Newelle for complex multi-step tasks. It is local-first, lightweight Python, supports Ollama, and ships shell/Python/browser/`computer` tools + subagents + MCP.
3. **Wayland control = agent-sh/computer-use-linux** as an MCP server, attached to both Newelle and gptme. It is the only candidate purpose-built for Wayland/GNOME GUI control (AT-SPI + GNOME Shell extension + portals + ydotool).

**Stage 2 — Wire the verified components:**
- Voice loop: use Newelle's built-in STT/TTS, but benchmark against **RealtimeVoiceChat** (which IS RealtimeSTT + RealtimeTTS + Ollama, your exact picks) and **GLaDOS** (600ms loop, MCP) as latency references; lift their turn-detection/interruption logic if Newelle's is weaker.
- TTS: Supertonic-3 on CPU (your pick), Kokoro-82M fallback (already used by Newelle/GLaDOS/Row-Bot/CAAL — low integration risk).
- Procedure RAG: model your SKILL.md hierarchy on **Fabric patterns** and **Newelle/QwenPaw Skills**; optionally use **AIChat** as a local RAG + OpenAI-compatible proxy in front of Ollama.

## Caveats
- **Star counts are rounded and source-dependent.** Where the research fetched live repo pages I used those; OpenCode (~147k April 2026 → ~174k some sources) and AIChat (~10k vs ~29k) show wide variance across third-party write-ups. Treat all counts as approximate and re-check at build time.
- **Discrepancies flagged vs your recorded data:** OpenCode org is **anomalyco** (ex-SST) — confirmed; Goose org is **aaif-goose** (moved to Agentic AI Foundation/Linux Foundation) — confirmed; OpenClaw ~378k stars — confirmed (genuinely that high). NyarchAssistant stars are ~386–399, slightly below your ~402. Open Interpreter license is cited as Apache-2.0 in your list but AGPL-3.0 appears in some docs — verify before redistributing.
- **"Qwen3.5-4B" is real** (Ollama `qwen3.5:4b`, Apache-2.0) but had **documented tool-calling bugs** in some Ollama versions — pin a known-good build and test tool calls before committing the architecture.
- **Seeva, Local-Voice, azibom** are below the threshold where building on them is wise; treat as references only. Seeva's MIT license and any local-model support were not re-confirmed from primary source.
- **Wayland is a genuine constraint for desktop control.** Under Wayland there is no global screen-scrape; capture/input go through PipeWire + xdg-desktop-portal (consent prompts) or AT-SPI/ydotool. This is exactly why computer-use-linux (portal/AT-SPI/ydotool-based) is the right brick and why X11-era OS-control approaches (Open Interpreter's OS Mode) are a poorer fit on GNOME/Ubuntu 26.04 Wayland.
- This analysis assumes your stated single-user, local-first, privacy-first goals; several cut projects (OpenClaw, QwenPaw, Moltis, LocalAGI) are excellent for *other* shapes (messaging-gateway or web-UI agents) and could be revisited if your vision shifts toward "reach me on my phone."
